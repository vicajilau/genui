# Architectural Proposal: Multi-Mode Chat (GenUI Playground)

This document describes the architecture for implementing multi-mode support in the Playground's chat. This will allow developers to evaluate the capabilities of Generative UI under three different execution topologies, comparing performance, latency, and instruction-following capabilities (A2UI JSON) across various models.

## 2. Detailed Mode Descriptions

### A. Serverless Mode (Direct API Client)
* **Inference:** Direct connection from the Flutter client to the Gemini API (or any other compatible provider) via HTTP Server-Sent Events (SSE).
* **Schemas:** Dynamically loaded in the system instruction using the in-memory `globalGenUISchemasPromptDescription` from RAM.
* **Use Case:** Self-contained client-side applications that do not require backend infrastructure and want to minimize initial network latency.

### B. Local Mode (On-Device LLM)
* **Inference:** 100% offline execution on the device's CPU/GPU using native bindings (e.g., `flutter_gemma` for Google models, or `llamadart` for GGUF models).
* **Schemas:** Loaded in memory using `globalGenUISchemasPromptDescription`.
* **Stress Test:** Serves to evaluate if lightweight models (Gemma 2B, Llama 3B, etc.) can strictly adhere to the A2UI JSON syntax structure and successfully render interfaces without breaking the layout.
* **Use Case:** Applications with extreme privacy requirements, offline usage, or zero inference costs.

### C. Server Mode (Remote Serverpod Backend)
* **Inference:** Orchestrated on the backend. Serverpod serves as a WebSocket real-time gateway connecting to cloud models (Gemini, Claude, GPT) or self-hosted backend models.
* **Communication:** Real-time bi-directional WebSockets to stream chat tokens and receive A2UI user interaction events.
* **Schemas:** The Serverpod server reads the static `genui_schemas.json` contract file (generated by Flutter's `build_runner`) at startup to dynamically register the available tools/functions for the LLM.
* **Use Case:** Production-scale applications that need to hide API keys, manage central database states, or run complex multi-agent orchestration logic before rendering the UI on the client.

## 3. Code Abstraction Layer (Flutter Client)

To keep the chat controller decoupled from the active inference engine, we propose defining a unified interface in the client:

```dart
abstract class ChatService {
  /// Initializes the connection or loads the local model weights.
  Future<void> initialize();

  /// Sends a prompt and returns a stream of text/JSON A2UI blocks.
  Stream<String> sendMessage({
    required String prompt,
    required List<Map<String, dynamic>> history,
  });
  
  /// Clears resources, controllers, and socket connections.
  void dispose();
}
```

### Concrete Implementations:
1. `GeminiServerlessChatService`: Contains the current client-side streaming logic using `Dio` and Server-Sent Events (SSE).
2. `LocalGemmaChatService`: Initializes Gemma on the local device thread and interacts with the model loaded in RAM.
3. `ServerpodChatService`: Employs the autogenerated Serverpod client to establish a WebSocket session for real-time bi-directional streaming.

## 4. Settings Panel Redesign (`SettingsDialog`)

The Playground app's settings panel will expand into a tabbed layout to configure the active connection, displaying only the parameters required for each mode:

* **Serverless Tab:** Input fields for the Gemini Developer API Key and model selection (e.g. Gemini 1.5 Flash vs Pro).
* **Local Tab:** Download/loading status of the model file (`.bin`) and sliders to tune local inference parameters (Temperature, Top-K, Top-P).
* **Server Tab:** Serverpod backend URL endpoint (`http://localhost:8080`) and authentication tokens.
